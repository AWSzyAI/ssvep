下面给你一版可直接拿去开工的“需求描述 + 实现方案”。我按一个真正要落地的工程来写，不按论文空话写。核心目标不是一次性把 Quest、双屏、YOLO、OpenBCI 全缝完，而是先把“刺激呈现层”做成一个可扩展底座，后面不同显示设备、不同运行环境、不同上层输入都能接进去。

一、项目目标

开发一个用于 SSVEP 刺激测试与后续交互验证的前端系统。系统需要能够在当前已有的 PC 双屏环境上运行，同时保留未来迁移到 Quest 等 XR 设备的能力。当前阶段的核心任务是稳定地产生四个不同频率的视觉刺激点或刺激块，并显示当前运行环境的屏幕参数，作为后续接入 YOLO 目标检测、FIFO 目标调度、EEG/BCI 解码、ROS/控制信号输出的基础界面。

系统的第一优先级不是视觉美观，而是刺激频率的可控性、不同刷新率设备上的适配能力、运行参数的可观测性，以及未来模块扩展时不用推翻重写。

二、当前已知运行环境约束

当前需要兼容三类显示环境。

第一类是笔记本内屏。已知参数为 2560×1600，165.02 Hz，支持可变刷新率，10-bit，RGB，SDR。它适合高刷新率测试，但 165.02 这种非整数刷新率会让“严格整除式频率设计”变得不优雅，所以更适合作为高性能调试环境，而不是“理论最干净”的基线环境。

第二类是外接显示器 Redmi 27 NU。已知当前工作模式是 3840×2160，60 Hz，8-bit，RGB，SDR，不支持 VRR，同时系统提供 75 Hz、71.93 Hz、70.07 Hz、50 Hz、29.56 Hz 等候选刷新率。这个屏幕更适合作为低刷基线验证环境，尤其是 60 Hz 和 75 Hz 这种更规则的模式。

第三类是未来的 Quest 设备。目标模式可以预设为 120 Hz。Quest 不仅是一个刷新率变化，更是渲染栈、输入方式、部署方式、全屏控制方式都不同的环境，因此不能把 Quest 当成“又一块显示器”，而应该从架构上把它看成一个独立适配层。

三、需求描述

1. 刺激显示需求

界面需要同时显示四个可独立配置频率的视觉刺激目标。当前阶段默认形态为圆点，也应预留切换为矩形块、边框块、目标覆盖块的能力。每个刺激目标需要在视觉上明确区分，并在其附近显示当前绑定的频率值。刺激应支持传统二值闪烁，也支持基于连续相位计算的亮度调制模式。

系统应支持全屏运行，以减少浏览器 UI、系统合成器、窗口边框等对刺激稳定性的影响。刺激位置在第一阶段可固定为四象限，后续应可扩展为任意坐标布局，以便覆盖 YOLO 检测出来的目标框。

2. 屏幕参数显示需求

界面左上角需要显示当前运行环境的关键参数。这里要明确分成两类信息：一类是浏览器可自动获取的实时参数，例如窗口大小、屏幕像素尺寸、devicePixelRatio、colorDepth、估计刷新率、当前全屏状态；另一类是需要人工配置或由宿主系统提供的设备档案参数，例如目标分辨率、标称刷新率、是否支持 VRR、HDR 状态、颜色格式、位深、屏幕名称、运行模式说明。系统必须把这两类信息区分开，避免把人工配置值误认为真实自动检测值。

3. 多显示设备适配需求

系统需要支持不同显示器档案的切换。最少需要内置以下几组预设：60 Hz 基线档、75 Hz 扩展档、165 Hz 高刷档、Quest 120 Hz 档。用户切换档案后，系统应自动切换推荐频率集合、刺激生成策略、参数展示内容。系统还应支持用户手动新建档案，以便未来添加其他显示器或头显。

4. 频率管理需求

系统不能把频率写死在代码里。必须有一层频率配置管理。对于不同设备，需要提供“推荐频率组”和“实验频率组”两类概念。推荐频率组用于当前屏幕上更稳定的刺激呈现；实验频率组用于和未来目标设备保持一致，例如当前在 165 Hz 内屏上预演 Quest 的 8/12/15/20 Hz。系统需要允许切换频率组，并实时更新界面与日志。

5. 运行模式需求

系统至少需要支持两种刺激模式。第一种是整帧模式，即按刷新帧周期进行二值开关，这适合规则刷新率屏幕上的基线验证。第二种是连续相位模式，即依据时间连续计算相位和亮度，再在每次渲染帧上取样输出，这更适合处理 165.02 Hz 这类不规则情况，也更贴近未来多平台扩展。系统需要能在两种模式间切换，并在参数面板中显示当前模式。

6. 可观测性与调试需求

系统必须具备基础调试能力，包括实时估计页面渲染刷新率、显示当前四个刺激频率、显示当前所用显示器档案、显示运行模式，并可选记录帧间隔分布、平均帧时、丢帧情况。原因很简单：SSVEP 系统里最容易出问题的不是 UI，而是“以为自己在 12 Hz，实际上出来的是脏频谱”。

7. 未来扩展需求

当前阶段不要求接入真实 EEG 解码，但架构上需要为以下模块留接口：YOLOv8 nano 目标检测结果输入、FIFO 候选目标管理、OpenBCI/EEG 解码结果输入、ROS 或其他控制系统输出、Quest/Unity/WebXR 设备适配。也就是说，刺激系统不能是一个只能显示四个死圆点的页面，而应该是一个“刺激引擎 + 布局层 + 设备档案层”的底座。

四、实现方案

1. 总体架构

建议把系统拆成六层。

第一层是设备档案层。负责描述当前运行设备的静态配置，例如设备名称、目标分辨率、标称刷新率、推荐频率组、颜色空间、位深、是否 Quest 模式、推荐刺激策略。这一层不要直接依赖浏览器 API，而是作为配置对象存在。这样未来从 PC 网页切到 Quest 或 Electron 时，上层逻辑可以不动。

第二层是运行时检测层。负责采集浏览器当前能拿到的动态信息，例如 `screen.width`、`screen.height`、`window.innerWidth`、`window.devicePixelRatio`、`screen.colorDepth`、`requestAnimationFrame` 推算出的实时刷新率、是否处于全屏等。这一层输出的是“观测值”，而不是“配置值”。

第三层是频率调度层。负责根据当前设备档案和实验目标选择四个刺激频率，并给每个刺激目标分配频率 ID、显示标签、颜色或编号。这一层后面也可以接 FIFO 逻辑，让新目标进来时自动占用某个频率槽位。

第四层是刺激引擎层。它是核心。负责根据运行模式输出每个刺激目标在当前时刻的可视状态。整帧模式下，它按离散帧开关；连续相位模式下，它依据 `sin(2πft + φ)` 计算亮度或二值阈值状态。这一层不应该关心刺激是圆点还是 YOLO 覆盖块，它只输出“这个目标此刻该亮还是该暗、亮度多少”。

第五层是布局渲染层。负责把刺激引擎输出映射到实际页面。初版可以是四象限固定位置的圆点，第二版可以变成矩形块，第三版可以直接覆盖在检测到的物体框上。因为刺激引擎和布局层解耦，所以后续接 YOLO 只是在布局层替换位置来源，不需要动频率逻辑。

第六层是外部接口层。负责接收检测结果、实验控制命令、BCI 解码输出、日志上传等。当前阶段可以先留空，或者用 mock 数据模拟。

2. 设备档案设计

建议每个设备档案至少包含这些字段：设备名称、目标分辨率、标称刷新率、可选刷新率说明、推荐频率组列表、推荐刺激模式、颜色格式、位深、颜色空间、HDR 状态、VRR 状态、备注。这样一来，你的内屏、外屏、Quest 都只是不同的 JSON 配置项。

例如，60 Hz 基线档可以定义为推荐频率 7.5、10、12、15。75 Hz 档可以定义为 7.5、9.375、12.5、15。165 Hz 稳定档可以定义为 11.79、12.69、13.75、15。Quest 120 Hz 档可以定义为 8、12、15、20。这样做的好处是，当前即便你没有 Quest，也可以先把 Quest 档案加入系统，在 PC 上用连续相位模式预演这组频率，上层实验逻辑以后就不用改。

3. 刺激频率策略

这里要明确一个原则：系统必须接受“不同设备上的频率策略不完全一致”这一现实。不要试图用一组四频打天下。60 Hz 屏最适合整帧规则频率，165.02 Hz 屏更适合连续相位近似，Quest 120 Hz 则适合另外一组干净整帧频率。也就是说，频率不是一个写死的常量，而是“设备相关的实验参数”。

对于当前双屏测试，建议同时保留两类频率组。第一类是设备最优组，用于验证当前显示器上的刺激稳定性；第二类是目标平台兼容组，用于提前模拟 Quest 未来的实验条件。这样开发路径会更顺：先证明“代码没问题”，再证明“迁移时逻辑一致”。

4. 渲染实现选择

当前阶段最适合用 Web 前端实现，因为你要先快速迭代和跨屏调试。页面渲染建议基于 `requestAnimationFrame`，不要用 `setInterval` 做频闪。后者在浏览器里会受事件循环、后台标签页、节流机制影响，稳定性太差。`requestAnimationFrame` 至少能与显示器刷新同步，尽管它依然不能保证物理输出绝对精确，但已经是前端里相对正确的路径。

视觉刺激建议支持两种输出方式。第一种是硬阈值二值闪烁，适合经典 SSVEP；第二种是正弦调制后再量化，适合调试和高刷屏近似。可以让这两种模式都保留，通过参数切换，而不是一开始就把路线锁死。

5. 页面布局设计

首页面建议左上角为参数监控面板，中央为刺激区域，右上角为控制面板。参数监控面板显示当前设备档案、自动检测值、当前四频、当前模式。刺激区域显示四个刺激目标。控制面板用于切换设备档案、频率组、渲染模式、全屏状态、目标形态。这个布局对开发最友好，因为所有关键变量都可见。等后续接入 YOLO 再把控制面板缩掉，切到实验模式。

6. YOLO/FIFO 扩展方式

你已经提到后续希望用 YOLOv8 nano 做目标检测，然后给物体打标记，并在目标上覆盖频闪块，还涉及 FIFO。这一部分不要一开始硬写进刺激页面。正确做法是把刺激目标抽象成一个统一结构，例如包含 `id`、`position`、`shape`、`size`、`frequency`、`label`、`visible` 等字段。当前阶段这四个目标的 `position` 是固定四象限；以后接 YOLO 后，`position` 就来自检测框中心或角点；FIFO 则负责决定哪些目标进入“活跃刺激集合”。这样一来，从静态圆点切到目标覆盖块，本质只是数据源变了，刺激引擎完全不用改。

7. Quest 可扩展性考虑

Quest 的扩展性主要体现在三点。第一，频率档案要能单独配置，不依赖当前 PC 屏。第二，渲染接口不能写死成“浏览器屏幕中心四圆点”，而应该支持相对于视口、相对于世界坐标、相对于物体锚点三种布局模式。第三，输入和部署方式要可替换。当前在 Web 上你用鼠标和按钮控制，以后到 Quest 可能是 gaze、controller、或 WebXR/Unity 事件驱动。所以现在就要把“控制逻辑”和“显示逻辑”拆开。这样未来即便从 HTML 页面迁到 Unity 或 WebXR，刺激调度和设备档案仍然能保留。

五、推荐开发路线

第一阶段只做 PC 网页版底座。实现设备档案切换、四个刺激目标、两种渲染模式、左上角参数显示、全屏控制、帧率估计。这个阶段的目标不是 BCI 成功，而是证明你的刺激生成器在不同屏幕上行为可控。

第二阶段做多屏实验验证。分别在 60 Hz 外屏、75 Hz 外屏、165 Hz 内屏上测试，记录各频率组的表现，比较整帧模式和连续相位模式的差异。这个阶段主要是建立你自己的“设备经验库”。

第三阶段做目标覆盖抽象。把固定四圆点改造成“可绑定坐标的四个刺激对象”，让它既可以显示在固定位置，也可以显示在外部输入的 bbox 上。这一步之后，YOLO 接入就很顺了。

第四阶段接入外部模块。先接 mock YOLO，再接真实 YOLOv8 nano，再接 FIFO，再接 EEG/解码结果输入。此时页面已经从“刺激演示器”进化成“视觉选择前端”。

第五阶段再考虑 Quest/WebXR/Unity。到这时你迁移的是宿主环境，不是重写系统。

六、关键技术判断

最重要的一点是，当前必须把“显示设备差异”当成一等公民，而不是异常情况。你的系统以后一定会面对 60、75、90、120、144、165 甚至带抖动的小数刷新率环境，所以从第一天开始，设备档案层、频率策略层、刺激引擎层就必须分开。

第二，网页能做前期开发，但它不是最终的物理校准工具。后面一旦进入正式实验，最好增加光电传感器或高速摄像验证实际输出频谱。现在先把架构搭对，比现在就陷入“精确到 0.02 Hz”的幻觉更重要。

第三，你当前没有 Quest，不是坏事。正好可以先把系统做成真正可迁移的底座，而不是为某个单一设备写死。

如果你要，我下一条可以直接继续输出两样东西之一：一份更正式的“PRD/需求文档版本”，或者一份“前端模块划分 + 数据结构定义 + 页面状态机”的技术设计稿。
